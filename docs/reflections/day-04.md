# Day 4 Reflection

## Why Raw Documents And Processed Events Are Separate

Raw documents preserve source truth. They store the original title, body, source, URL, timestamp, hash, and ingestion metadata.

Processed events are derived intelligence. They store the normalized market event that the platform can summarize, classify, search, cluster, and display.

Keeping them separate matters because AI enrichment can be wrong. If an event summary, sentiment label, or entity extraction is bad, the platform can trace it back to the raw source and reprocess it later.

## What Should Be Stored Raw

Store the original source material and ingestion facts:

- source name
- source URL
- original title
- original body
- published timestamp
- content hash
- source-specific metadata

This data should remain close to what entered the system.

## What Should Be Stored As Derived Intelligence

Store structured outputs separately:

- normalized financial events
- summaries
- topics
- sentiment labels
- entities
- embeddings
- narrative clusters
- job status and errors

These fields are useful to the product, but they are generated by the system and may need to be recalculated.

## Why This Matters For Debugging AI Pipelines

AI systems need traceability. When a generated answer is weak or a narrative cluster looks wrong, the system should answer:

- Which raw document produced this event?
- Which enrichment job processed it?
- Which entities were extracted?
- Which embedding was used for retrieval?
- Which events supported the narrative or answer?

The schema creates that audit path from source data to user-facing intelligence.

## Day 4 Decision

Use SQLAlchemy models for the domain schema and Alembic migrations for database history. Store embeddings as JSON for the initial schema so tests and development do not require pgvector immediately. The project will migrate this to pgvector-backed vector columns when semantic search is implemented.
